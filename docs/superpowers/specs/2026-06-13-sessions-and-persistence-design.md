# C1 — Sessions & Persistence Design Spec

**Date:** 2026-06-13  
**Status:** Approved  
**Depends on:** Nothing (foundational)  
**Required by:** C2 (Answer Quality), C3 (Chat UX)

## Problem

LocalMind has zero conversation persistence. Messages live in React `useState`. Refreshing the page, switching "chats", or changing browsers destroys all history. There is no session list, no switching, no rename, no export. This makes the app unusable as a real knowledge base tool.

## Solution

Backend-owned session persistence using SQLite. Full message objects (role, content, sources, latency, model) stored server-side. Frontend fetches sessions and messages via REST API. Session titles auto-generated by LLM after the first Q&A exchange. Document selection and model preference remembered per-session.

## Database Schema

File: `./localmind.db` (alongside `./chroma_db/`)

```sql
CREATE TABLE IF NOT EXISTS sessions (
    id          TEXT PRIMARY KEY,
    title       TEXT NOT NULL DEFAULT 'New Chat',
    doc_ids     TEXT DEFAULT '[]',
    model       TEXT,
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS messages (
    id          TEXT PRIMARY KEY,
    session_id  TEXT NOT NULL,
    role        TEXT NOT NULL CHECK(role IN ('user', 'assistant')),
    content     TEXT NOT NULL,
    sources     TEXT,
    latency_ms  INTEGER,
    model       TEXT,
    created_at  TEXT NOT NULL,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_messages_session_id
    ON messages(session_id);
```

**`sessions` columns:**
- `id` — `uuid4().hex[:12]`, same format as doc_ids
- `title` — defaults to `"New Chat"`, updated by auto-title generator
- `doc_ids` — JSON array string, e.g. `["abc123", "def456"]`. Empty array `[]` means "all docs"
- `model` — preferred model ID (e.g. `meta/llama-3.1-8b-instruct`). NULL means "use server default"
- `created_at` / `updated_at` — ISO 8601 UTC timestamps

**`messages` columns:**
- `id` — `uuid4().hex[:12]`
- `session_id` — FK to sessions, cascading delete
- `role` — `"user"` or `"assistant"` (CHECK constraint)
- `content` — the message text (full markdown for assistant messages)
- `sources` — JSON array of SourceChunk objects (assistant messages only, NULL for user)
- `latency_ms` — response time in ms (assistant messages only, NULL for user)
- `model` — which model generated this (assistant messages only, NULL for user)
- `created_at` — ISO 8601 UTC timestamp, used for message ordering

## API Endpoints

### POST /sessions
Create a new session.

**Request body:** (all optional)
```json
{
  "title": "New Chat",
  "model": "meta/llama-3.1-8b-instruct",
  "doc_ids": ["abc123"]
}
```

**Response:** `201 Created`
```json
{
  "id": "a1b2c3d4e5f6",
  "title": "New Chat",
  "model": "meta/llama-3.1-8b-instruct",
  "doc_ids": ["abc123"],
  "created_at": "2026-06-13T10:00:00Z",
  "updated_at": "2026-06-13T10:00:00Z",
  "messages": []
}
```

### GET /sessions
List all sessions, ordered by `updated_at DESC`. Lightweight — no messages, just metadata.

**Response:** `200`
```json
{
  "sessions": [
    {
      "id": "a1b2c3d4e5f6",
      "title": "RAG Architecture Explained",
      "updated_at": "2026-06-13T10:30:00Z",
      "message_count": 4
    }
  ]
}
```

### GET /sessions/{id}
Get a session with all its messages ordered by `created_at ASC`.

**Response:** `200`
```json
{
  "id": "a1b2c3d4e5f6",
  "title": "RAG Architecture Explained",
  "model": "meta/llama-3.1-8b-instruct",
  "doc_ids": ["abc123"],
  "created_at": "2026-06-13T10:00:00Z",
  "updated_at": "2026-06-13T10:30:00Z",
  "messages": [
    {
      "id": "msg001",
      "role": "user",
      "content": "What is RAG?",
      "sources": null,
      "latency_ms": null,
      "model": null,
      "created_at": "2026-06-13T10:00:05Z"
    },
    {
      "id": "msg002",
      "role": "assistant",
      "content": "## Retrieval-Augmented Generation\n\nRAG is...",
      "sources": [{"doc_id": "abc123", "filename": "doc.pdf", "chunk_index": 0, "content": "...", "score": 0.92}],
      "latency_ms": 1840,
      "model": "meta/llama-3.1-8b-instruct",
      "created_at": "2026-06-13T10:00:07Z"
    }
  ]
}
```

### PATCH /sessions/{id}
Update session metadata. Any subset of fields may be provided.

**Request body:** (all optional)
```json
{
  "title": "My Custom Title",
  "doc_ids": ["abc123", "def456"],
  "model": "meta/llama-3.1-70b-instruct"
}
```

**Response:** `200` — updated session object (same shape as POST response)

### DELETE /sessions/{id}
Delete a session and all its messages (CASCADE).

**Response:** `200`
```json
{ "status": "deleted", "id": "a1b2c3d4e5f6" }
```

### POST /sessions/{id}/messages
Save a message to a session. Also triggers auto-title generation if this is the first assistant message and the title is still `"New Chat"`.

**Request body:**
```json
{
  "role": "assistant",
  "content": "## Retrieval-Augmented Generation\n\nRAG is...",
  "sources": [{"doc_id": "abc123", "filename": "doc.pdf", "chunk_index": 0, "content": "...", "score": 0.92}],
  "latency_ms": 1840,
  "model": "meta/llama-3.1-8b-instruct"
}
```

**Response:** `201 Created`
```json
{
  "id": "msg002",
  "session_id": "a1b2c3d4e5f6",
  "role": "assistant",
  "content": "## Retrieval-Augmented Generation\n\nRAG is...",
  "sources": [...],
  "latency_ms": 1840,
  "model": "meta/llama-3.1-8b-instruct",
  "created_at": "2026-06-13T10:00:07Z",
  "auto_title": "RAG Architecture Explained"
}
```

The `auto_title` field is only present when the title was auto-generated. The frontend uses this to update the session list without a refetch.

## Auto-Title Generation

Triggered when:
1. A message with `role: "assistant"` is saved via `POST /sessions/{id}/messages`
2. AND the session currently has 0 existing messages (this is the first exchange)
3. AND the session title is still `"New Chat"`

Process:
1. Call the LLM with a dedicated title prompt
2. Update `sessions.title` in the database
3. Return the new title in the response as `auto_title`

Prompt:
```
System: "Summarize the following Q&A in 3-5 words. Output only the title, no quotes, no punctuation."
User: {user_question}
Assistant: {assistant_response_truncated_to_500_chars}
```

LLM config: same NVIDIA NIM client, `max_tokens=20`, `temperature=0.3`, uses the session's model (or server default).

If title generation fails (LLM error, timeout), the session retains `"New Chat"` title — no error surfaced to user.

## Frontend Changes

### App.jsx

Current state:
```jsx
const [chatKey, setChatKey] = useState(0);
```

New state:
```jsx
const [sessions, setSessions] = useState([]);
const [currentSessionId, setCurrentSessionId] = useState(null);
```

**On mount:** `GET /sessions` → populate `sessions` list. If sessions exist, select the most recent. If none, create one via `POST /sessions`.

**"New Chat" button:** `POST /sessions` → get new session, prepend to `sessions`, set as `currentSessionId`.

**Session switching:** Click session in sidebar → `setCurrentSessionId(id)`.

**Model change:** When user selects a different model, `PATCH /sessions/{currentSessionId}` with `{model: newModel}`. Update local `sessions` array.

**Doc selection change:** When user toggles document checkboxes, `PATCH /sessions/{currentSessionId}` with `{doc_ids: newIds}`. Debounced 500ms to avoid spamming on rapid toggles.

### Sidebar.jsx

New layout order (top to bottom):
1. Brand logo + theme toggle + close (existing)
2. **"new --chat" button** (existing, now creates session via API)
3. **Sessions section** (NEW)
   - Header: `// sessions (N)` in mono font
   - List of session items, each showing:
     - Title (truncated, 1 line)
     - Relative time: `2h ago`, `yesterday`, `Jun 10`
     - Active session highlighted with `bg-accent/5 border-accent/20`
   - Hover actions: rename (pencil icon), delete (trash icon)
   - Rename: inline text input replaces title, Enter to save, Esc to cancel
   - Delete: `window.confirm()` then `DELETE /sessions/{id}`, remove from list
4. File uploader (existing)
5. Sources section (existing)
6. System status (existing)

### ChatWindow.jsx

**On `sessionId` change:** `GET /sessions/{id}` → populate `messages` state from response. Apply session's `doc_ids` and `model` to parent state via callbacks.

**After streaming completes (`onDone` callback):**
1. `POST /sessions/{sessionId}/messages` — save user message `{role: "user", content: question}`
2. `POST /sessions/{sessionId}/messages` — save assistant message `{role: "assistant", content, sources, latency_ms, model}`
3. If response contains `auto_title`, update session title in parent `sessions` array + sidebar

**Streaming flow unchanged:** The actual query/streaming still goes through `/query/stream`. Persistence happens after completion only.

## New Backend Files

### `backend/database.py`
- `init_db()` — creates tables if not exist, called on app startup
- `get_db()` — returns sqlite3 connection (thread-safe via `check_same_thread=False`)
- Session CRUD functions: `create_session()`, `get_sessions()`, `get_session()`, `update_session()`, `delete_session()`
- Message CRUD functions: `save_message()`, `get_messages()`

### `backend/routes/sessions.py`
- FastAPI router with all 6 endpoints
- Calls `database.py` functions
- `POST /sessions/{id}/messages` triggers title generation via `title_generator.py`

### `backend/llm/title_generator.py`
- `generate_title(question, answer, model)` → `str`
- Uses same NVIDIA NIM client as generator.py
- Short prompt, `max_tokens=20`, returns title string
- Returns `None` on failure (caller keeps default title)

### `backend/models/session_schemas.py`
- Pydantic models: `SessionCreate`, `SessionResponse`, `SessionListItem`, `SessionListResponse`, `MessageCreate`, `MessageResponse`, `SessionDetail`, `SessionUpdate`, `DeleteSessionResponse`

## Migration & Compatibility

- **No data migration needed** — this is a new feature, existing users have no sessions
- SQLite DB auto-created on first startup via `init_db()`
- Existing `/query` and `/query/stream` endpoints unchanged — they remain stateless
- Frontend gracefully handles empty session list (auto-creates first session)

## Error Handling

| Scenario | Behavior |
|---|---|
| SQLite unavailable | App startup fails with clear error |
| Session not found (GET/PATCH/DELETE) | `404 Not Found` |
| Message save fails | `500`, frontend shows error toast, message lost (user can retry) |
| Title generation fails | Silently keep `"New Chat"`, log warning |
| Session list fetch fails | Frontend shows empty list, creates new session |

## Constraints

- SQLite connection uses `check_same_thread=False` (FastAPI runs in thread pool)
- `doc_ids` JSON serialization/deserialization handled in `database.py`
- `sources` JSON serialization/deserialization handled in `database.py`
- All timestamps are ISO 8601 UTC
- Session `updated_at` bumped to `now()` on every `save_message()` call
- DELETE uses CASCADE — no orphaned messages
